- PR polling not adaptive ‚Äî **PARTIALLY DEBUNKED / RECLASSIFIED** ‚Äî poll-pr-checks.mjs is explicitly NON-blocking: ONE `gh pr checks --json` call per invocation (poll-pr-checks.mjs:54), exits 0 always, no `--watch` (header :5-9, :17-27). HOWEVER the scheduler re-emits a separate `poll-ci` action for EVERY `awaiting-ci` issue every tick (scheduler.mjs:46-48) with no batching (each is its own `gh` subprocess) and no per-PR backoff ‚Äî pacing is delegated entirely to the orchestrator ("ScheduleWakeup ~4 min", SKILL.md:111). - Wave packing not proportional ‚Äî **CONFIRMED** ‚Äî wave-plan.mjs:15-43 `planWaves` is exactly greedy file-disjoint bin-packing: sort by issue number (:17), then for each wave walk `remaining` and admit an issue only if none of its `targetFiles` is already in `usedFiles` (:23-30), capped at `maxWaveSize`. No affinity/grouping, no balancing ‚Äî purely first-fit-disjoint by ascending issue number. Confirmed as described; low-value optimization. ‚Äî target: Phase 5 (deferrable)
- Refute-gate vs missing refute field ‚Äî **RECLASSIFIED ‚Üí DEBUNKED as a swarm correctness bug (benign)** ‚Äî the cross-skill premise is half-true but misapplied. merge-pr.mjs reads `refuteVerdict` from a function PARAMETER / CLI `--refute-verdict` arg (merge-pr.mjs:16,47) and imports NO ledger (only execFileSync, merge-pr.mjs:7) ‚Äî it never reads the upstream-merge ledger at all, so that ledger lacking a refute field is irrelevant to the merge gate. In the SWARM path the verdict is carried by the swarm's OWN ledger, which DOES have a `refute` field (swarm-ledger.mjs:64), written on the `refute-pending ‚Üí approved` transition (e.g. `recordTransition(path,n,"approved",{refute:{tally:{panelVerdict:"approve"}}})`, tested at __tests__/integration/happy-path.test.mjs:28). The scheduler only emits a `merge-pr` action for issues in state `approved` (scheduler.mjs:64-65), and `approved` is reachable ONLY from `refute-pending` (swarm-ledger.mjs:21) ‚Äî so by construction every merge has already passed the refute panel. SKILL.md:114 hardcodes `--refute-verdict approve` precisely because the merge action is gated on the `approved` state; auto-merge does NOT always block. The fail-safe at merge-pr.mjs:21-22 (refuteVerdict===null ‚Üí blockedBy:"refute-missing") is real defense-in-depth but is never exercised on the swarm path because the literal `approve` is always supplied. The one residual smell (low severity): the literal `approve` is not read back from `ledger.issues[n].refute.tally.panelVerdict`, so an orchestrator that emitted `merge-pr` out of band could pass `approve` without a real verdict ‚Äî but the state machine prevents that today. Not a Phase 3 correctness bug. ‚Äî target: Phase 5 (optional: make merge read the recorded ledger verdict instead of a hardcoded literal)

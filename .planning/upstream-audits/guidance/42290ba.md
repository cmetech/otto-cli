# 42290ba — fix: preserve marker identity with repo metadata

## Target file(s)
- src/resources/extensions/workflow/repo-identity.ts

## Divergence
This is a one-line follow-up to 3f5e830 that broadens the marker-identity reuse condition from `hasProjectState(markerPath)` to `hasProjectState(markerPath) || readRepoMeta(markerPath)`. It applies inside the `if (remoteUrl === null) { ... markerPath ... }` recovery block that 3f5e830 introduces. Otto's `repo-identity.ts` exports `readRepoMeta` (line ~46 in current otto) and has `gsdHome`, `process.env.GSD_STATE_DIR`, and `readGsdIdMarker` available — so all the supporting infra is present once 3f5e830 lands. Without 3f5e830 there is no `if (remoteUrl === null)` block to broaden; this commit is meaningless on its own.

## Concrete edits
1. After porting 3f5e830, locate the new block inside `repoIdentity`:
   ```ts
   if (remoteUrl === null) {
     const markerRoot = resolveGitRoot(basePath);
     const markerId = readGsdIdMarker(markerRoot);
     if (markerId) {
       const markerPath = join(process.env.GSD_STATE_DIR || gsdHome(), "projects", markerId);
       if (hasProjectState(markerPath)) return markerId;
     }
   }
   ```
   and change the inner predicate to `if (hasProjectState(markerPath) || readRepoMeta(markerPath)) return markerId;`.
2. No new tests are added by the upstream commit; nothing else to port.

## Verdict
Worth porting as a tail of 3f5e830. It closes a small race where a freshly-created external state dir has metadata written but no project state yet, which would otherwise cause the identity to bounce. Don't port in isolation — it depends on 3f5e830's recovery block existing. Apply both together.

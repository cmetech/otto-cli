# 42ce989 — feat(coding-agent): hyperlink file paths in tool titles

## Target file(s)
upstream: `packages/coding-agent/src/core/tools/{edit,find,grep,ls,read,write,render-utils}.ts`
otto-cli candidates: `packages/pi-coding-agent/src/core/tools/{edit,find,grep,ls,read,write}.ts` — but `render-utils.ts` DOES NOT EXIST in otto-cli.

## Divergence
The entire tool-rendering layer this feature builds on has been restructured in otto-cli — a cherry-pick is impossible and a hand-port would require rebuilding the feature against a different architecture:
- No `render-utils.ts` in `packages/pi-coding-agent/src/core/tools/` (confirmed by directory listing). Therefore no `shortenPath`, `invalidArgText`, `str`, or the new `linkPath`/`renderToolPath` helpers the feature adds.
- otto-cli's tool files use a different rendering model. e.g. `read.ts` imports `AgentTool` from `@otto/pi-agent-core` and `tool-target.ts` (`createReadFileTarget`, `ToolTargetMetadata`); it has none of the upstream `formatReadCall`/theme-based `theme.fg("accent", ...)` title-rendering functions the patch edits. There is no `formatReadCall`, `formatEditCall`, `formatLsCall`, `formatWriteCall`, or `renderCall(args, theme, context)` signature to modify.
- The feature depends on `getCapabilities().hyperlinks` + `hyperlink()` from pi-tui wrapping theme-styled title text — otto-cli's tool titles are produced by a different (target-metadata-driven) path.

## Concrete edits
Not portable as a diff. Re-implementing the feature in otto-cli would mean: locating where otto-cli renders tool-call titles for read/write/edit/ls (likely via `tool-target.ts` / pi-agent-core), adding an OSC 8 `file://` wrap (absolute path target, relative/home-shortened visible text, line annotations kept out of the target) gated on `getCapabilities().hyperlinks`. This is a new feature build, not a port — out of scope for a cherry-pick audit. Defer unless explicitly prioritized.

## Verdict
do-not-port — depends on `render-utils.ts` and a theme-based title-render architecture that otto-cli does not have; tools are restructured around `tool-target.ts`/pi-agent-core. Would require a from-scratch reimplementation rather than a port.

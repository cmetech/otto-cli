# 491ace3 — fix(pi-ai): remove duplicate response id normalization

## Target file(s)
- packages/pi-ai/src/providers/openai-responses-shared.ts

## Divergence
gsd-pi deletes two lines from `convertResponsesMessages`:
```
- } else if (msgId.length > 64) {
-   msgId = `msg_${shortHash(msgId)}`;
}
```
The motivation: a later `uniqueResponseItemId(msgId)` already handles the >64-char truncation, so the inner branch is redundant. Otto's `openai-responses-shared.ts` (line 167-184) still contains the exact same redundant branch (lines 173-175 in otto):
```ts
let msgId = parsedSignature?.id;
if (!msgId) {
  msgId = `msg_${msgIndex}`;
} else if (msgId.length > 64) {
  msgId = `msg_${shortHash(msgId)}`;
}
```
There are two small contextual divergences:
1. gsd-pi's pre-fix first branch is `msg_${msgIndex}_${textItemIndex}` (with a `textItemIndex += 1` follow-up); otto's is `msg_${msgIndex}`. Different cosmetic choice, no impact on the patch.
2. gsd-pi wraps the result through `msgId = uniqueResponseItemId(msgId);` after the `if/else if`; otto does NOT have a `uniqueResponseItemId` call here. *That is the crux.* If otto has no later normalization, deleting the `else if` branch removes the >64-char protection entirely and ships malformed ids to OpenAI.

## Concrete edits
1. Grep otto's `openai-responses-shared.ts` (and `openai-responses.ts`, `openai-shared.ts`) for `uniqueResponseItemId`, `shortHash`, and the 64-character rule. If a downstream normalizer exists that enforces the 64-char limit, delete the `else if (msgId.length > 64) { msgId = "msg_" + shortHash(msgId); }` block (lines ~174-176 in otto).
2. If no downstream normalizer exists, do not delete — instead, leave a comment noting that gsd-pi consolidated this through `uniqueResponseItemId` and that otto retains the inline guard.

## Verdict
Worth porting *conditionally*. The fix is a real cleanup (duplicate normalization risks two different transformations diverging), but otto's openai-responses pipeline diverged at exactly the seam that makes the upstream cleanup safe: without `uniqueResponseItemId` downstream the inline guard is the only thing keeping ids ≤ 64 chars.

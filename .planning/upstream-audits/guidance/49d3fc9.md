# 49d3fc9 — fix(gsd): resolve worktree registry root from checkout

## Target file(s)
- src/resources/extensions/workflow/repository-registry.ts
- src/resources/extensions/workflow/tests/repository-registry.test.ts

## Divergence
Otto's `repository-registry.ts` *already includes* the new helper `resolveGitWorkingTreeRoot` (line 57) and uses it in `createRepositoryRegistry`:
```ts
const projectRoot = resolveGitWorkingTreeRoot(contract.workRoot) ?? contract.projectRoot;
```
But the gsd-pi commit guards this with `contract.isWorktree ? resolveGitWorkingTreeRoot(contract.workRoot) ?? contract.projectRoot : contract.projectRoot`. Otto's version invokes the helper *unconditionally* for every registry construction, which is broader than the gsd-pi fix and changes behaviour for non-worktree project roots (running a synchronous `git rev-parse --show-toplevel` on every registry call). Either:

(a) Otto deliberately broadened the fix and the unconditional call is intentional — confirm by reading any otto-side commit message that introduced the helper. In that case, no port needed.

(b) Otto landed an early draft without the `contract.isWorktree` gate — port the gate to avoid the extra git call and to keep behaviour predictable in non-worktree mode.

The companion test `tests/repository-registry.test.ts` adds an "external-state worktree checkout as project root" case. Otto's test file likely needs the same case added.

## Concrete edits

2. (Option b) Change `const projectRoot = resolveGitWorkingTreeRoot(contract.workRoot) ?? contract.projectRoot;` to `const projectRoot = contract.isWorktree ? resolveGitWorkingTreeRoot(contract.workRoot) ?? contract.projectRoot : contract.projectRoot;`. Verify `contract.isWorktree` is exposed by `resolveGsdPathContract` — gsd-pi exports it; otto's may need a small additional export.

Treat as a cleanup / behaviour-pin port rather than a critical fix; the bug being addressed (worktree registry pointing at the wrong root) is already mitigated in otto.

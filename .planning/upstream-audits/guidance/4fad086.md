# 4fad086 — fix(ci): allow build-native to publish engine packages at a target semver

## Target file(s)
- .github/workflows/build-native.yml
- scripts/publish-engine-packages.sh (does not exist in otto-cli)

## Divergence
Diverged. Otto has its own `.github/workflows/build-native.yml` that already publishes `@cmetech/otto-engine-*` packages. It currently reads the version from `package.json` directly (`VERSION=$(node -p "require('./package.json').version")`) and has no `platform_version` workflow_dispatch input. Upstream's fix adds a `platform_version` input so production releases can publish engines at a bumped semver before main is updated. Otto does NOT have `scripts/publish-engine-packages.sh` — the publish loop is inlined in the workflow yaml. The same underlying release-coordination problem applies to otto (you need a way to publish engine packages at a bumped version before the root release commit lands), but the fix has to be re-shaped against otto's inlined publish step.

## Concrete edits
1. In `.github/workflows/build-native.yml` add a new `platform_version` workflow_dispatch input alongside `publish` and `publish_auth`:
   ```yaml
   platform_version:
     description: "Override engine package semver (defaults to package.json version)"
     required: false
     default: ""
     type: string
   ```
2. In the publish job, replace the unconditional `VERSION=$(node -p "require('./package.json').version")` with: prefer `inputs.platform_version` when non-empty; else fall back to package.json.
3. Apply that `VERSION` to the `npm version --no-git-tag-version $VERSION` / `npm publish` step inside the per-platform loop (lines ~258 / ~283 / ~297 — the three `@cmetech/otto-engine-${platform}` references).
4. Skip the `scripts/publish-engine-packages.sh` half of the patch — otto has no such script; the equivalent logic is the inlined publish loop.
5. Verify `scripts/__tests__/build-native-workflow.test.mjs` exists in otto (it likely does given prior slices) and update assertions to expect the new input.

## Verdict
Manual-port. The CI-coordination need is real for otto's release flow too (per memory: `otto release + publish flow` uses tag-driven build-native.yml → npm-publish.yml), but the surface differs enough that this is a port-the-idea exercise, not a cherry-pick. Low-risk change once mapped — pure CI plumbing, no runtime code touched.

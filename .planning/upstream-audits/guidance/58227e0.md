# 58227e0 — fix(bug-2): Crash logs use unbounded per-call filenames

## Target file(s)
- src/resources/extensions/workflow/bootstrap/crash-log.ts
- src/resources/extensions/workflow/tests/crash-handler-secondary.test.ts

## Divergence
Near-identical. Otto's `bootstrap/crash-log.ts` is functionally the same as upstream's (uses `OTTO_HOME` and `~/.otto/crash/` instead of `GSD_HOME` / `~/.gsd/crash/`, with `[otto]` prefix instead of `[gsd]`). The bug — generating a new timestamped log file per call so a crash loop floods the crash directory — exists identically in otto (verified: otto's crash-log.ts uses `new Date().toISOString().replace(/[:.]/g, "-")` then `join(crashDir, \`${ts}.log\`)`). Otto also has `crash-handler-secondary.test.ts` at the same logical path, ready to receive the new test case.

## Concrete edits
1. In `src/resources/extensions/workflow/bootstrap/crash-log.ts`:
   - Update the header comment to read `~/.otto/crash/pid-<pid>.log` instead of `<timestamp>.log`.
   - Replace the two-line timestamp filename block with a single line:
     ```ts
     const logPath = join(crashDir, `pid-${process.pid}.log`);
     ```
   - Remove the `const ts = ...` line above it.
2. In `src/resources/extensions/workflow/tests/crash-handler-secondary.test.ts`:
   - Port the new `'writeCrashLog appends repeated crashes from one process to a single file'` test case verbatim, except:
     - Replace `GSD_HOME` → `OTTO_HOME` in env get/set/restore.
     - Replace the `gsd-crash-test-` tmpdir prefix with `otto-crash-test-`.
     - Keep the assertion `logs.length === 1` and the `first crash` / `second crash` content checks unchanged.

## Verdict
Cherry-pick. Same bug, same fix, only s/gsd/otto/ on the env var and log prefix. CRITICAL_STABILITY severity is correct — uncaught/unhandled crash loops in production would fill the user's disk under the old logic. Port and ship in the next patch release.

# 71369f9 — fix: gate all stranded work during auto bootstrap

## Target file(s)
- src/resources/extensions/workflow/auto-start.ts
- src/resources/extensions/workflow/tests/auto-start-orphan-bootstrap.test.ts (new or extend)

## Divergence
Otto's `auto-start.ts` exports `auditOrphanedMilestoneBranches` at line 245 and calls it during `bootstrapAutoSession` at line 898 — same shape as upstream. We were unable to grep `isBlockingStrandedWorkAction`, `strandedRecoveryActions`, or `blockingStrandedWork` in otto's auto-start, so the structure is pre-patch (single `strandedRecoveryAction` rather than the array `strandedRecoveryActions`). The bug: when multiple milestones have in-progress stranded work, only the first blocking action gets considered, so a different stranded milestone (not the one matching `state.activeMilestone`) can be silently ignored. Upstream fixes this by collecting *all* blocking actions and preferring the one that does not match the active milestone.

## Concrete edits
1. In `src/resources/extensions/workflow/auto-start.ts`, add the helper `isBlockingStrandedWorkAction(action)` near `OrphanAuditResult`.
2. Replace single-action find with `strandedRecoveryActions = auditResult.actions.filter(isBlockingStrandedWorkAction)`. Keep `strandedRecoveryAction = strandedRecoveryActions[0] ?? null` for backwards compat.
3. After computing `state.activeMilestone`, compute `blockingStrandedRecoveryAction = state.activeMilestone ? strandedRecoveryActions.find(a => a.milestoneId !== state.activeMilestone?.id) ?? strandedRecoveryAction : strandedRecoveryAction;` and use it for the two error notifies + `return releaseLockAndReturn()` blocks.
4. Update the journal/log payload to include `strandedRecoveryActions` array as well.
5. Add a test that exercises two milestones with stranded work + an active milestone matching one of them, asserting the *other* is reported as the blocker.

## Verdict
manual-port — the bug and the fix are both data-shape changes that translate 1:1 to otto. Function names and types align; only the auto-start.ts file is touched. High value (correctness for users with multiple stranded milestones); low risk.

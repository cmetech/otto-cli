# 9b7f522 — fix(gsd): log repo identity remote failures

## Target file(s)
- src/resources/extensions/workflow/repo-identity.ts

## Divergence
Otto already has `workflow-logger.ts` exporting `logWarning` (and an audit/test suite around it), so the new dependency exists. However otto's `getRemoteUrl` currently swallows ALL errors silently (returns `""` on any catch) — there is no `console.warn` to replace. Upstream's pre-patch state used `console.warn`; otto skipped that warning entirely. So the port is not a 1:1 string substitution: we must add the logging branch from scratch and reconcile that otto's signature returns `string` (not `string | null`).

## Concrete edits
1. In `src/resources/extensions/workflow/repo-identity.ts` add `import { logWarning } from "./workflow-logger.js";` near the top.
2. In `getRemoteUrl(basePath)` (around line 190), expand the `catch` block:
   - If the error's `code` is `ENOENT` (no git binary) or stderr mentions `does not have any commits yet` / `not a git repository`, return `""` silently as today.
   - For any other error, call `logWarning("state", "repo-identity: failed to resolve remote.origin.url", { basePath, error: String(error) })` then return `""`.
3. Keep the return type as `string` (not change to `string | null`) to avoid cascading typecheck failures at the two call sites (lines 293, 585).

## Verdict
Manual-port because otto already silently swallowed the failure case the upstream patch was trying to surface, so the patch text does not apply directly. The intent (distinguish transient git failures from "no remote configured" via structured logging) translates cleanly given otto already owns `workflow-logger.ts`. Take care not to log warnings for projects that are simply not git repos — those should remain silent.

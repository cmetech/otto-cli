# a36a132 — fix(ai): abort Codex SSE body reads

## Target file(s)
`packages/pi-ai/src/providers/openai-codex-responses.ts` (test: `packages/pi-ai/test/openai-codex-stream.test.ts` if present)

## Divergence
The same code path exists but has diverged in signatures, so a clean cherry-pick will NOT apply:
- otto-cli `processStream` (line 446) is `processStream(response, output, stream, model)` — it does NOT accept an `options` param, and calls `processResponsesStream(mapCodexEvents(parseSSE(response)), output, stream, model)` (line 452). Upstream threads `options` and passes a service-tier options object to `processResponsesStream`; otto-cli's `processResponsesStream` call has a different (shorter) shape. Do NOT copy upstream's service-tier options block.
- otto-cli `parseSSE(response)` (line 498) takes no `signal` param — identical in body to upstream's pre-fix version (reader/decoder/buffer loop with a `finally { reader.cancel() }`).
- otto-cli already aborts at the outer level: the IIFE checks `options?.signal?.aborted` after `processStream` and sets `stopReason = "aborted"` (lines 338-345). But the per-read abort inside parseSSE (the actual fix) is missing — a long SSE body read won't be interrupted mid-stream.

## Concrete edits
1. Change `parseSSE(response: Response)` → `parseSSE(response: Response, signal?: AbortSignal)`.
2. Inside parseSSE: add `const onAbort = () => { void reader.cancel().catch(() => {}); }; signal?.addEventListener("abort", onAbort, { once: true });` after the reader is created.
3. In the `while (true)` loop, add `if (signal?.aborted) throw new Error("Request was aborted");` both before and after `await reader.read()`.
4. In the `finally`, add `signal?.removeEventListener("abort", onAbort);` before the existing `reader.cancel()`.
5. Thread the signal to the call site: change `processStream`'s call at line 452 to `parseSSE(response, ...signal...)`. NOTE the signal source differs — otto-cli's `processStream` has no `options` param. Either add an `options?` param to `processStream` (matching the upstream shape) OR pass the signal that's already in scope in the IIFE down to `processStream`/`parseSSE`. Verify how the signal reaches line 452 before applying; do NOT blindly add upstream's `options?.serviceTier`/`resolveServiceTier` block (otto-cli's processResponsesStream call doesn't take it).

## Verdict
manual-port — logic applies cleanly conceptually, but signatures diverged (no `options` on processStream, `processResponsesStream` call shape differs). Port the parseSSE signal-abort changes by hand and wire the signal through otto-cli's actual call chain.

# b70e549 — fix: wait for workspace packages after publish

## Target file(s)
- /Users/coreyellis/code/github.com/cmetech/otto_app/otto-cli/.github/workflows/build-native.yml

## Divergence
The patch hardens the post-publish step in `.github/workflows/build-native.yml` by adding a `wait_for_workspace_package` shell function that polls `npm view <pkg>@<version>` up to five times with backoff, called immediately after each `npm publish --workspace ${workspace}` so the next step doesn't race on npm-registry propagation. Otto has `.github/workflows/build-native.yml` at the same path. The upstream workspace list iterated by the original loop is `@opengsd/contracts @opengsd/rpc-client @opengsd/mcp-server` — those are gsd-pi's workspace package names. Otto's workspace names use `@otto/...` (or possibly `@cmetech/...`) and the set may differ (otto packages: contracts, rpc-client, mcp-server are all present). The shell logic is otto-neutral but the workspace list literal needs to match otto's actual workspace publish loop.

## Concrete edits
1. Open `.github/workflows/build-native.yml` in otto.
2. Locate the `for workspace in ...; do npm publish --workspace ${workspace} ...; done` loop in the publish-workspace job.
3. Above the loop, define the `wait_for_workspace_package()` shell function exactly as in upstream (lines 309–332 of the upstream file), parameterized on `$1`/`${PACKAGE}` and using `${VERSION}` from the surrounding step's env. The loop should: `for attempt in $(seq 1 5)`, query `npm view "${PACKAGE}@${VERSION}" version`, return 0 on match, sleep DELAY (starting at 5s, doubling, capped at 30s), and `exit 1` after attempt 5 with a `::error::` annotation.
4. After each `npm publish --workspace "${workspace}" ...` line inside the loop, add `wait_for_workspace_package "${workspace}"`.

## Verdict
Manual-port. The race condition the fix addresses — the main package publish stepping on a not-yet-visible workspace package after `npm publish --workspace` succeeds — applies to any monorepo that publishes workspaces serially. Otto has the same shape of workflow. The shell function is mechanical to copy; only the literal workspace package names differ. Caveat: confirm otto's release docs (UPSTREAM-PIPELINE runbook) acknowledge the wait step before adding it, in case otto's release flow expects an immediate next-step start.

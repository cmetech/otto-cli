# c100620 — fix(coding-agent): Clean up Path Handling (#4873)

## Target file(s)
Upstream centralizes everything onto packages/coding-agent/src/utils/paths.ts. In otto-cli, src/utils/paths.ts DOES NOT EXIST (and neither does src/utils/child-process.ts that paths.ts imports). The 23 touched files (config.ts, agent-session*.ts, auth-storage.ts, export-html/index.ts, extensions/loader.ts, model-registry.ts, package-manager.ts, prompt-templates.ts, resource-loader.ts, sdk.ts, session-manager.ts, settings-manager.ts, skills.ts, tools/path-utils.ts, main.ts) exist under packages/pi-coding-agent/ but none route through a `utils/paths.ts` module.

## Divergence
Architecturally incompatible. The commit's core move is: introduce `normalizePath`/`resolvePath`/`PathInputOptions` in utils/paths.ts and rewrite tools/path-utils.ts + 21 callers to delegate to it. otto-cli's tools/path-utils.ts still carries the pre-refactor inline logic (`UNICODE_SPACES`, `normalizeUnicodeSpaces`, `normalizeAtPrefix`, `os.homedir()` expansion in `expandPath`/`resolveToCwd`) and has diverged further (e.g. `normalizeMsysPath`, an extra ` ` in its UNICODE_SPACES regex). Every pre-image would conflict, and the central module the diff edits is absent.

## Concrete edits
No mechanical port. A faithful port would mean reimplementing the entire `utils/paths.ts` consolidation and re-threading 20+ otto-cli callers — effectively redoing the refactor, with high regression risk against otto-cli's already-customized path handling (msys/windows tweaks). Not worth porting as a unit. If a specific path-handling bug from this PR is identified, fix it pointwise in otto-cli's existing tools/path-utils.ts.

## Verdict
do-not-port — depends on a src/utils/paths.ts module otto-cli does not have; large cross-cutting refactor against already-diverged, platform-customized path code.

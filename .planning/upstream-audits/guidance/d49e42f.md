# d49e42f — fix(packaging): resolve undici after npm global install

## Target file(s)
- scripts/link-workspace-packages.cjs
- scripts/validate-pack.js
- package.json

## Divergence
Upstream addresses an `npm install -g` regression: the bundled `@gsd/pi-coding-agent` imports `undici`, but a global install leaves an empty `node_modules/undici` placeholder at the root. The fix: (1) add an `ensureRootUndici()` helper to `scripts/link-workspace-packages.cjs` that seeds root `node_modules/undici` from `packages/pi-coding-agent/node_modules/undici` via symlink+cpSync fallback, (2) extend `scripts/validate-pack.js` with an import-smoke that does `await import(...http-dispatcher.js)` plus a full `npm install -g` smoke that asserts the global root has a resolved `undici/package.json`, (3) align the root `undici` dep to `8.3.0` and bundle it. Otto-cli is structurally similar — `packages/pi-coding-agent` is vendored, `scripts/link-workspace-packages.cjs` exists and uses the same symlink-with-cp-fallback pattern, `scripts/validate-pack.js` exists and runs install + smoke. Differences: otto root pins `undici: "^7.24.2"` (not `8.3.0`), otto's package namespace is `@otto/pi-coding-agent` (not `@gsd/...`), and otto's global install path under `lib/node_modules/...` uses `@cmetech/otto` not `@opengsd/gsd-pi`.

## Concrete edits
1. In `scripts/link-workspace-packages.cjs`, add the `ensureRootUndici()` helper (mirror upstream lines 24–58 verbatim) but make sure `REPO_ROOT` already imports correctly (it does, via `./lib/workspace-manifest.cjs`). After the existing `for (const name of ['pi-agent-core', 'pi-ai', 'pi-tui', 'pi-coding-agent'])` block, call `if (ensureRootUndici()) process.stderr.write('  Linked undici from shipped pi-coding-agent bundle\n')`.
2. In `scripts/validate-pack.js`, replicate the http-dispatcher import smoke against `node_modules/@otto/pi-coding-agent/dist/core/http-dispatcher.js` (rename scope from `@gsd` to `@otto`). Add the global-install smoke block but rename the expected globalRoot to `join(globalPrefix, 'lib', 'node_modules', '@cmetech', 'otto')`.
3. Decide on the `undici` version. Upstream chose 8.3.0; otto sits on `^7.24.2`. If `@otto/pi-coding-agent`'s http-dispatcher works on 7.x in otto today, keep 7.x and just add the placeholder-resolution + smoke. If not, bump to 8.x but check `package-lock.json` for transitive callers first.
4. Confirm root `package.json#bundleDependencies` or `bundledDependencies` list includes `undici` so the tarball ships it (otto currently bundles only `xlsx`). Adding `undici` here is the structural fix for the placeholder bug.

## Verdict
Manual port. The bug class is real for otto too: any user running `npm i -g @cmetech/otto` hits the same `undici` placeholder hazard because the symptom is npm's global install layout, not gsd-specific. The diff will not apply cleanly (paths + scope + undici version differ), and the riskiest piece is the bundling/version decision — leave that to a human who can run the global-install smoke locally and pick the right undici major. The `ensureRootUndici` helper itself is mechanically portable.

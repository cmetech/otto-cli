# e9146a5 — fix(coding-agent): use async operations in tools

## Target file(s)
packages/pi-coding-agent/src/core/tools/{bash,edit,find,grep,ls,read,write,path-utils}.ts, src/cli/file-processor.ts, src/utils/image-resize.ts (and upstream-new src/core/tools/file-mutation-queue.ts, src/utils/image-resize-core.ts, src/utils/image-resize-worker.ts — none exist in otto-cli)

## Divergence
Architecturally incompatible on the two largest fronts:
- Image pipeline: upstream is sharp-based and this commit adds a Worker-thread split (image-resize-core.ts + image-resize-worker.ts). otto-cli's image-resize.ts is already a different implementation built on photon (`SamplingFilter.Lanczos3`, async `resizeImage`/`tryBothFormats`). The upstream Worker refactor does not map; the new files have no otto-cli home.
- file-mutation-queue.ts does not exist in otto-cli (no equivalent module).
- The tools (edit/read/write) in otto-cli already do not use the sync-fs calls upstream is replacing (grep shows 0 `readFileSync`/`writeFileSync` in edit/write/read), so the core sync→async conversion is largely moot here, and otto-cli has additional divergence (hashline edit/read variants not present upstream).
None of the per-file pre-images can be expected to match.

## Concrete edits
No mechanical port. If async-I/O correctness in a specific otto-cli tool is later found lacking, address it as a targeted standalone fix against otto-cli's actual (photon/hashline-based) implementation — do not attempt to import upstream's Worker/queue architecture.

## Verdict
do-not-port — large multi-file refactor against an image/tool architecture otto-cli has already replaced; cherry-pick would conflict everywhere and the new modules have no equivalent.
